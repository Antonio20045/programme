# Datenbank-Integration

- Tools: Factory Pattern — `createXxxTool(userId, pool/oauth)` statt globale Instanzen.
- Notes/Reminders: Per-User via `createNotesTool(userId, pool)` / `createRemindersInstance(userId, pool)`.
- Gmail/Calendar: Per-User OAuth via `GoogleOAuthContext` (Tokens aus `user_oauth_tokens`, AES-256-GCM verschlüsselt).
- Agent Memory: Per-Agent per-User via `agent-memory.ts` — isolierter Key-Value-Store (JSONB). Funktionen: `set/get/getAll/getByCategory/deleteKey/deleteNamespace/cleanupExpired/cleanupStaleCache/formatForSystemPrompt`. TTL per Kategorie (preference=∞, learned=90d, state=1d, cache=explizit). Migration: 005.
- Agent Registry: Per-User via `agent-registry.ts` — CRUD für Sub-Agent-Definitionen (max 20 pro User). Funktionen: `createAgent/getAgent/getUserAgents/getActiveAgents/updateAgent/updateStatus/touchAgent/deleteAgent`. ID: kebab-case Name + 6 Hex. Migration: 004. Lokal (SQLite) deaktiviert wie Notes/Reminders.
- Pattern Tracker: Standalone-Utility `pattern-tracker.ts` (kein AgentTool). Erkennt wiederholt ähnliche User-Anfragen und schlägt Agent-Erstellung vor. Funktionen: `trackRequest/checkForPattern/dismissSuggestion/markCreated/cleanupOld`. Max 200 Patterns/User, Threshold 3 in 14 Tagen, Cooldown 30 Tage. `cleanupOld` ist global (nur Admin/Cron). Migration: 006.
- Agent Executor: `executeAgent(task, pool, llmClient)` in `agent-executor.ts` — führt Sub-Agents als isolierte LLM-Calls aus. Nutzt `getAgent`/`touchAgent` (registry), `formatForSystemPrompt` (memory), `resolveModelForAgent` (model-resolver). LLM-Zugriff via injiziertem `LlmClient` Interface. Keine eigene DB-Migration.
- Delegate Tool: `createDelegateTool(userId, pool, llmClient)` in `delegate-tool.ts` — Factory-Pattern, wrappet `executeAgent()` als `ExtendedAgentTool`. Validiert agentId (kebab-case, max 100), task (max 10K), context (optional, max 5K). Prüft Agent-Status (`active` required). Mappt `AgentResult`-Status (success/partial/failure/needs-approval) auf strukturiertes JSON. Sanitized Error-Handling (keine internen Details nach außen). Keine eigene DB-Migration.
- Orchestrator Classifier: `classify(message, userId, pool)` in `orchestrator-classifier.ts` — regelbasierte Nachrichtenklassifikation für Sub-Agent-Routing. Kein LLM-Call. Nutzt `getActiveAgents` (registry) für Agent-Matching, `trackRequest` (pattern-tracker) fire-and-forget. Heuristiken (hasMultipleSubTasks, requestsAnalysis, isCodingTask, requiresMultiToolCoordination) gespiegelt aus `gateway/src/model-router.ts` — keep in sync. Output: `ClassificationResult` (complexity, category, matchedAgents, modelTier, parallelExecution). Keine eigene DB-Migration.
- Budget Controller: `budget-controller.ts` — Utility-Modul (kein Tool) für Per-Agent Tagesbudgets. `checkBudget(pool, agentId, userId) → BudgetStatus` prüft ob Agent unter Tageslimit (100K Input, 50K Output, 100 Tool-Calls). `recordUsage(pool, agentId, userId, usage)` UPSERT (additive pro Tag). `getDailyStats(pool, agentId, userId, days=7)` historische Stats. `resetExpired(pool, maxAgeDays=30)` Cleanup (Cron). Input-Validierung: agentId (kebab-case 2-100), userId (non-empty, max 200), usage (non-negative int), days (positive int 1-3650). SQL: `make_interval(days => $N::int)` statt String-Concat. TOCTOU-Caveat dokumentiert. Migration: 007.
- Agent Lifecycle: `agent-lifecycle.ts` — Utility-Modul für automatische Agent-Bereinigung (kein Tool). `runLifecycleCheck(pool, userId)` prüft `lastUsedAt` aller Agents: >180d → `deleteNamespace` + `deleteAgent`, >90d → `updateStatus('archived')` + Memory-Cleanup (alles außer `preference`), >30d → `updateStatus('dormant')` + `cronSchedule` nullen. `reactivateAgent(pool, userId, agentId)` setzt dormant/archived → active + `touchAgent`. `runMemoryCleanup(pool)` global: `cleanupExpired` + `cleanupStaleCache(7)`. Keine eigene DB-Migration.
- Agent Factory: `createAgentFactoryTool(userId, pool)` in `agent-factory.ts` — Factory-Pattern, wrappet Agent-Erstellung als `ExtendedAgentTool`. Validiert Name (1-50 Zeichen, Allowlist, case-insensitive Duplikat-Check via `getActiveAgents`), Tools (Existenz-Check via `getTool()`), Cron (5-Feld, min 15 Min, semantische Range-Validierung), Model (nur haiku/sonnet). Leitet `riskProfile` ab via `getToolRiskTier()`. Baut System-Prompt. Erstellt Agent via `createAgent(pool, userId, input)`. Keine eigene DB-Migration.
- Tool-Factory: `createUserTools(userId, pool, llmClient?)` erstellt alle User-Tools. Gmail/Calendar nur wenn OAuth-Provider konfiguriert und Tokens vorhanden. Delegate + Agent-Factory nur wenn `llmClient` gesetzt (= PostgreSQL-Modus).
- LlmClient Adapter: `createLlmClient()` in `gateway/src/llm-client-adapter.ts` — wrappet pi-ai `completeSimple()` im `LlmClient`-Interface. Lazy Model-Discovery-Cache. Singleton-Pattern.
